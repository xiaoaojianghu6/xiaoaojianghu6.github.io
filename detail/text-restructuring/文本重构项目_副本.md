# 文本重构项目：基于语言学理论的逻辑结构化引擎

## 一、现有工具的局限与本项目定位

长段表述普遍存在四类问题：结构混乱、逻辑顺序不清晰、宕开一笔后遗漏要点、随口提及的重要内容被忽视。

**知识管理工具（Obsidian/Heptabase/Logseq）**：聚焦手动构建知识网络，不提供自动化的逻辑标注、结构重组和矛盾检测。**论证工具（Argdown）**：需手动转写为专用语法，仅覆盖论证关系，不涉及修辞结构和言语行为分析。**AI 写作助手（Copilot/Notion AI）**：侧重生成内容，面对已写好的长文倾向概括摘要而非全量结构化。**直接 LLM 梳理**：信息压缩、幻觉填补、无系统性矛盾检测、无增量状态追踪。

**本项目定位**：全自动的、基于语言学理论的、保留全部细节的逻辑结构化重建管线——同时主动发现矛盾/遗漏/模糊概念并以非强制方式提问。

## 二、理论到实现的映射

每项核心机制有明确的语言学/逻辑学理论支撑。

### 2.1 修辞结构理论 (RST) -- Mann & Thompson 1988

定义文本单元的角色（Nucleus/Satellite）和修辞关系（Cause/Result/Concession/Evidence/Background/Elaboration/Condition/Purpose/Contrast/Sequence）。

**实现**：Tagger 输出 `role` + `relation` 字段，GraphUpdater 映射为 19 种 EdgeType 有向边。不做教条应用：`relation=none` 不强制降级 satellite；scope 推理采用质量检查触发而非数量检查。

### 2.2 言语行为理论 (Speech Act Theory) -- Searle 1969

五类言语行为：断言(Assertive)、指令(Directive)、承诺(Commissive)、表达(Expressive)、宣告(Declaration)。

**实现**：Tagger 输出 `intent` 字段（五分类）。对抗机中 Prosecutor 根据 intent 差异化搜索证据（assertive->incoming_evidence, directive->incoming_background, commissive->future_reference）。EntityHedgeScanner 利用 commissive 类型做承诺追踪。

### 2.3 Dung 论证框架 -- Dung 1995

论证图中的 Attack/Defense 关系，可接受性通过语义扩展判定。

**实现**：Tagger 输出 `argument_role`(premise/claim) + `argument_relation`(support/attack)，映射为 SUPPORT/ATTACK 边。ConfidenceManager 的攻防辩论（成功+0.1，失败-0.15，最多 2 轮）直接对应 Dung 框架的论证博弈。

### 2.4 艾宾浩斯遗忘曲线 -- Ebbinghaus 1885

对数衰减规律：R = e^(-t/S)。

**实现**：DecayEngine 对图谱节点做 `weight = exp(-t * decay_rate)` 衰减。三种速率：foreshadow(0.01, 极慢)、多线叙事(按 thread_id 差异化)、标准。ActiveBuffer 维护 Top-3 活跃节点，衰减至阈值以下入池；后续引用可"打捞"并重新激活——对应复习效应。

### 2.5 AGM 信念修正 -- Alchourron/Gardenfors/Makinson 1985

理性信念变更的三类操作（扩充/收缩/修正）和最小变更原则。

**实现**：ConfidenceManager 初始置信度 1.0。Prosecutor 起诉 = 修正请求，Defender 辩护 = 抵抗修正。增量非对称（成功+0.1/失败-0.15）。硬阈值 2 轮辩论上限体现最小变更原则——超限直接推入 AlertBuffer 交人类判定。

### 2.6 因果图模型 + Allen 时序逻辑

CAUSE/RESULT 边对应因果图模型，SEQUENCE 边对应 Allen(1983) 的 BEFORE/AFTER。

## 三、管线架构

```python
输入 -> SentenceSplitter -> Tagger(流式Mini-Batch+ActiveBuffer)
     -> GraphUpdater(增量图谱+四组件对抗)
     -> VPGA Refiner(全局重写) -> FCR验证 -> 输出
```

LLM 调用：Tagger(N/8 批) + SemanticRouter(最多 5 对审计) + VPGA(1 次)，8000 字约 15-25 次。

### 3.1 Tagger：Mini-Batch LLM 标注

每 8 句一批发送 LLM，每句输出 10 字段（text/scope/role/relation/intent/dependency/argument_role/argument_relation/thread_id/anchor_type）。Prompt 注入 ActiveBuffer 的 Top-3 活跃节点摘要，实现跨距离上下文感知。CLI 支持流式模式（`batch_tag_streaming`），API 管线（`/api/audit`）使用批量模式（`batch_tag`），均为同一 LLM 标注逻辑。

### 3.2 增量语义图谱

7 条纯 Python 建边规则（零 LLM）：

| 规则 | 触发条件 | 边类型 |
|------|----------|--------|
| 1 | chunk.relation 非 none | 映射 EdgeType (cause/result/evidence 等) |
| 2 | chunk.argument_relation | ATTACK / SUPPORT |
| 3 | 同 scope + chunk_index 差 < 5 | CO_SCOPE (双向弱连接) |
| 4 | chunk.dependency 非空 | DEPENDS -> BACKGROUND |
| 5 | thread_id/anchor_type | THREAD_BRIDGE / REVEAL (多线叙事) |
| 6 | 代词标记 | 图谱回溯 CO_SCOPE (不污染 scope 字段) |
| 7 | ActiveBuffer | 节点添加 + tick 衰减 |

**代词消解**：Tagger 标记 `_pronoun_needs_resolution=True` 后，GraphUpdater 反向搜索图谱找最近非代词节点建立隐式绑定，消解结果不入 scope。

**多线叙事**：节点带 `thread_id`/`anchor_type`，跨 thread 建 THREAD_BRIDGE（线索关联）和 REVEAL（后续揭示伏笔）。

**跨文件增量**：图谱持久化（save/load），节点记录 `file_id`/`file_path`，文件夹处理时累积语义状态。

### 3.3 对抗验证系统

对抗机周期在 GraphUpdater.process() 末尾执行，四组件并行产出 alert，合并输出（updater.py:276）：

```python
return conf_manager.get_alerts() + semantic_alerts + entity_hedge_alerts + cross_file_alerts
```

#### 组件 A：AdversaryScanner + 对抗辩论（基础层）

**AdversaryScanner**：7 项纯 Python 规则检查（LOW_CONFIDENCE/ISOLATED_NODE/LOGIC_CONFLICT/DANGLING_DIRECTIVE/STRUCTURE_CHAOS/UNRESOLVED_PRONOUN/ARGUMENT_GAP）。

**Prosecutor-Defender-ConfidenceManager**：Scanner 产出 raw_alerts -> Prosecutor 构建结构化起诉（含言语行为增强搜索）-> 按 node_id 分组 -> Defender 逐起诉辩护 -> ConfidenceManager 更新置信度，低于 0.7 入 AlertBuffer。

Defender 的 11 种搜索策略：

| 策略 | 搜索目标 |
|------|----------|
| incoming_support / incoming_evidence / incoming_background / incoming_cause / incoming_condition | 各类型入边 |
| co_scope_neighbors | CO_SCOPE 邻居的支持边 |
| same_scope | 同 scope 非孤立节点 |
| text_match | 4 字符关键词匹配（过滤虚词，优先同 scope） |
| future_reference | 后续节点引用当前节点 |
| outgoing_support | 出站 SUPPORT 边 |
| same_scope_support | 同 scope >= 3 条 SUPPORT 边 |

#### 组件 B：SemanticRouter（双轨混合对抗）

**轨道 1 -- Python 拓扑粗筛**：扫描 2 种可疑模式：(a) 同 scope 活跃节点间无逻辑边；(b) directive 节点缺 CAUSE/CONDITION/BACKGROUND 入边（悬空指令）。

**轨道 2 -- LLM 语义审计**：对 Top-5 可疑节点对单次 LLM 调用审计，LLM 仅需回答"SAFE"或指出问题核心。代价从 O(N^2) 降至 O(5)。

#### 组件 C：EntityHedgeScanner（实体对冲，零 LLM）

三个纯 Python 子检测：

1. **跨距离实体对冲**：仅追踪高精度类型（quoted_term 如「项目预算」、url），排除 `[一-鿿]{2,4}` 滑动窗口产生的碎片。同实体出现在距离 >= 3 且不同 scope 时触发。

2. **数值冲突**：同 scope 下显式冲突（同量词比例 > 2.0）+ 隐含约束（如人均最低成本规则，"10 万元预算雇佣 50 人"）。

3. **承诺追踪**：检测承诺性语句在后续文本中是否有执行描述，用滑动窗口匹配上下文核心名词。

#### 组件 D：CrossFileConflictDetector（跨文件冲突，LLM 驱动）

numeric_extractor 正则提取 + 簇分组两轮合并 -> 批量 LLM 审计同一实体的跨文件矛盾 -> 漏检回溯（LLM 指出缺失实体时从图谱二次搜索）。

### 3.4 VPGA 全局重写引擎

与传统管线不同，VPGA 直接发送原文 + 14 条约束规则给 LLM 做一次性全局重写，不依赖标注质量。双模式：writer（保留文风）/ developer（精简高效）。temperature=0.15, max_tokens=8192。

后处理：`extract_and_clean()` 提取 [疑点:] 和 [发现的问题:] 标记，分离阅读文本和提问列表。`RefinementRefiner` 支持增量微调——用户回答问题后秒级修改而非全量重跑。

### 3.5 FCR 事实覆盖率

客观量化指标，不依赖 LLM 主观评分。detail_anchor.py 提取四类锚定实体（URL/数值含单位/比喻标记/引号术语），在重构文本中精确字符串匹配：`FCR = preserved / total`。
